# Prompt 22 — Generate AGENT-HANDOFF-PROTOCOL.md

## Role
You are an orchestration protocol agent. You produce AGENT-HANDOFF-PROTOCOL.md — Document #22 in the 24-document SDLC stack (Full-Stack-First approach). This document defines HOW the 24 document-generating agents communicate, pass context, validate outputs, and handle failures. Without this, agents are isolated scripts. With it, they form a coordinated pipeline.

## Why This Document Exists

The 24 documents (00-23) define WHAT to build. But they are generated BY agents — one agent per document. These agents need a contract for:
- How to pass one document's output as the next agent's input
- Where documents are stored between steps
- How to validate an output before passing it downstream
- What happens when an agent fails or produces low-quality output
- How the orchestrator knows which step to run next

This is the **meta-protocol** — the protocol that governs the document generation pipeline itself.

## Input Required
- Correct Build Sequence.md (dependency graph — which agent depends on which)
- ARCH.md (session store, database, infrastructure)
- CLAUDE.md (naming conventions, repo structure)
- QUALITY.md (quality rubric for document scoring)

2. **Session Context Store** — How documents are passed between agents:

   ```python
   # Session context is a key-value store scoped to a pipeline run

   class SessionStore:
       """Persists document outputs between agent steps."""

       async def write(self, run_id: str, key: str, value: str) -> None:
           """Store a document. Key is from the Agent Registry 'Session Write Key'."""
           # Stores to: PostgreSQL session_context table
           # Schema: (run_id, key, value, written_by, written_at, version)

       async def read(self, run_id: str, key: str) -> str | None:
           """Read a document. Returns None if not yet written."""

       async def read_many(self, run_id: str, keys: list[str]) -> dict[str, str]:
           """Read multiple documents. Raises if any required key is missing."""

       async def exists(self, run_id: str, key: str) -> bool:
           """Check if a document has been written."""
   ```

   **Storage rules:**
   - Every agent reads from session keys listed in "Session Read Keys"
   - Every agent writes exactly ONE session key listed in "Session Write Key"
   - Session values are the full document content (markdown or JSON string)
   - Session is scoped to a `run_id` (one pipeline execution)
   - Session persists across restarts (PostgreSQL-backed, not in-memory)

4. **Orchestrator Protocol** — How the pipeline controller decides what to run:

   ```
   ORCHESTRATOR ALGORITHM:

   1. Load dependency graph from Agent Registry
   2. For each step in topological order:
      a. Check: Are ALL "Depends On" agents completed with passing quality scores?
         - If NO: Wait (or fail if a dependency failed)
         - If YES: Proceed
      b. Resolve inputs: Read all "Session Read Keys" from SessionStore
         - If any key is missing: FAIL (dependency tracking bug)
      c. Build message envelope with resolved inputs
      d. Execute agent:
         - Send prompt + inputs to LLM
         - Receive document output
         - Validate output (see Quality Gate below)
         - Store output to SessionStore under "Session Write Key"
      e. Log: agent_id, duration, cost, quality_score, status

   3. Parallel execution rules:
      - Steps 1-2 (ROADMAP + PRD): Can run in parallel (both read raw_spec + brd_doc)
      - Steps 8-9 (MCP-TOOL-SPEC + DESIGN-SPEC): Can run in parallel (both read interaction_map)
      - Steps 17-18 (MIGRATION-PLAN + TESTING): Can run in parallel (independent inputs)
      - All other steps are sequential

   4. Total pipeline cost ceiling: $45 per full run
   ```

5. **Quality Gate** — Validation between every step:

   ```
   QUALITY GATE PROTOCOL:

   After agent produces output, BEFORE writing to SessionStore:

   1. FORMAT CHECK (fast, no LLM):
      - Markdown documents: Has all required ## sections from prompt template?
      - JSON documents: Valid JSON? Matches expected schema?
      - IDs present: Does it use I-NNN, F-NNN, Q-NNN, S-NNN formats correctly?
      - Cross-references valid: Do referenced IDs from upstream documents exist?

   2. RUBRIC SCORING (uses LLM as judge):
      - Score document against quality rubric from QUALITY.md (Section 10)
      - Dimensions: completeness, accuracy, format_compliance, cross_interface_parity, traceability
      - Weighted average must be >= 0.85 to pass

   3. DECISION:
      - PASS (>= 0.85): Write to SessionStore, proceed to next step
      - CONDITIONAL (0.75-0.84): Write to SessionStore with warning, flag for human review
      - FAIL (< 0.75): Do NOT write. Retry (max 2 retries with feedback). Escalate after 2 failures.

   4. RETRY PROTOCOL:
      - On retry, include quality gate feedback in the agent's prompt:
        "Previous attempt scored 0.72. Failed dimensions: completeness (missing Section 4),
         traceability (F-003 references non-existent C8). Fix these specific issues."
      - Retry uses same session inputs (upstream docs haven't changed)
      - Max 2 retries. After that: pause pipeline, notify human, write partial output for inspection.
   ```

6. **Failure & Recovery Modes**:

   | Failure Type | Detection | Recovery |
   |-------------|-----------|----------|
   | Agent timeout (> 5 min) | Orchestrator timer | Kill, retry once with shorter max_tokens |
   | Quality gate fail (< 0.75) | Rubric scorer | Retry with feedback (max 2x), then escalate |
   | Missing input key | SessionStore.read returns None | Bug — dependency graph is wrong. Halt and alert. |
   | LLM API error (rate limit) | HTTP 429 | Exponential backoff: 10s → 30s → 60s → fail |
   | LLM API error (server) | HTTP 500/503 | Retry 3x with 5s delay, then fail step |
   | Cost ceiling breached | CostTracker check after each step | Pause pipeline, notify human with cost report |
   | Circular dependency detected | Topological sort fails | Halt. This is a build sequence bug. Fix the dependency graph. |
   | Partial pipeline (resume) | SessionStore has some keys populated | Skip completed steps (keys exist + quality passed), resume from first missing key |

### Quality Criteria
- Every agent has exactly ONE write key and knows its read keys
- Orchestrator algorithm handles the parallel steps correctly
- Quality gate rubric matches QUALITY.md Section 10
- Failure modes cover: timeout, quality fail, API error, cost breach, resume
- Audit trail captures everything needed for debugging and cost analysis
- Pipeline can resume from any step (checkpoint/resume)

### Anti-Patterns to Avoid
- Agents reading from each other directly (must go through SessionStore)
- No quality gate between steps (garbage in → garbage out cascade)
- Hard-coded step order (use dependency graph for topological sort)
- No retry with feedback (blind retries don't improve quality)
- No cost ceiling (runaway LLM spend)
- No audit trail (can't debug failures or optimize costs)
- Agents that write multiple session keys (violates single-responsibility)
